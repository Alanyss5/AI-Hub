# AI-Hub Windows 内部正式使用门槛

## 当前判断
AI-Hub 现在已经达到“Windows 自用可投入使用”，但还没有达到“Windows 内部正式使用候选”。

阻塞项已经不在基础功能，而在安装验证、长时间运行、签名与受控升级演练。

## P0：进入内部正式使用前必须完成

### 1. 发布与安装
- 固化真实可复现的 `win-x64 self-contained` 发布链路
- 固化 Inno Setup 编译环境，并跑通安装 / 升级 / 卸载 smoke
- 持续保留升级前自动备份 `config/ projects/ mcp/` 等本地状态
- 明确升级失败后的回滚路径并完成实测

### 2. 诊断与恢复
- 保持全局未处理异常捕获、启动失败日志、日志轮转和诊断包导出可用
- 给托管 MCP、脚本执行、Skills 同步、配置导入导出持续补统一错误码 / 错误分类
- 把诊断导出与最近一次启动失败入口纳入安装后 smoke

### 3. 安全与信任边界
- 保持脚本执行、托管 MCP 启动、外部 MCP 纳管的首次确认 / 风险确认
- 持续校验日志、通知、诊断导出中的敏感环境变量、token、路径脱敏
- 明确 AI-Hub 的写文件边界、进程启动边界和 PowerShell 触发边界

### 4. 数据可靠性
- 保持 JSON/TOML 存储的 schema version 与迁移入口可用
- 保持配置导入、Hub root 切换、MCP 写回、Skills 合并的“先备份再写入”策略
- 补齐权限不足、目录缺失、磁盘不可写、目标文件被占用时的完整实测

### 5. 发布前验证
- 增加打包后 smoke：首次启动、托盘隐藏 / 恢复、单实例、Skills 操作、MCP 体检与同步、配置包导入导出、升级保留原数据
- 增加长时间运行验证：后台维护循环、通知去重、托管 MCP 自恢复
- 固化发布前检查清单，不再依赖人工记忆

### 6. 治理与支持
- 持续维护第三方依赖清单与 Notices
- 补安装、升级、回滚、常见故障、自检步骤、日志位置的运维手册
- 明确内部支持流程和故障上报入口

## 完成定义
只有当 P0 六项全部落地并完成验证，README 与路线图口径才能从“Windows 自用可投入使用”升级为“Windows 内部正式使用候选”。
