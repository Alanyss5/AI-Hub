# 文档维护约定

## 核心原则

AI-Hub 文档采用“边开发边维护”的方式，不采用“全部功能完成后再集中补文档”的方式。

原因：

- 本项目同时涉及目录结构、脚本兼容层、配置文件、桌面界面和多客户端行为
- 如果延后补文档，很容易忘记真实行为、遗漏边界条件、写出过时说明

## 哪些变化必须同步更新文档

出现以下任意一种变化时，都应该在同一轮迭代内更新文档：

- 新增或删除一个用户可见按钮、页面或流程
- 修改全局级 / 项目级的行为
- 修改 Profile 的含义、默认值或适用范围
- 修改目录结构、配置文件路径或生成文件位置
- 修改脚本入口或从脚本迁移为程序内逻辑
- 新增一个常见故障场景
- 新增一个用户需要感知的重要限制
- 新增一个会影响后续协作方式的流程规则、交接规则或复用经验

## 最低更新要求

### 如果改了用户可见流程

至少更新：

- `AI-Hub-使用手册.md`
- `AI-Hub-版本变更.md`

### 如果改了目录或入口

至少更新：

- `README.md`
- `docs/README.md`
- `AI-Hub-快速开始.md`
- `AI-Hub-版本变更.md`

### 如果新增故障点

至少更新：

- `AI-Hub-故障排查.md`
- `AI-Hub-版本变更.md`

### 如果新增流程问题、交接规则或可复用经验

至少更新：

- `AI-Hub-开发路线图.md`
- `AI-Hub-工作流沉淀与Skill固化指南.md`

## 推荐工作流

1. 先实现功能
2. 功能一旦能跑通，立即补对应文档
3. 构建或运行验证通过后，再结束这轮迭代
4. 如果本轮只修了阻塞问题，结束前必须把主线未完成项写回路线图
5. 如果本轮沉淀出新的协作经验，结束前必须把经验写回工作流沉淀文档

## 文档分类

当前建议按下面的分类维护：

- `README.md`：仓库入口和当前状态
- `docs/README.md`：文档索引
- `AI-Hub-快速开始.md`：首次使用和最短路径
- `AI-Hub-使用手册.md`：稳定功能说明
- `AI-Hub-故障排查.md`：问题定位
- `AI-Hub-版本变更.md`：用户可感知变化
- `AI-Hub-开发路线图.md`：范围、状态和会话交接控制
- `AI-Hub-工作流沉淀与Skill固化指南.md`：工作流问题、修正方案和 Skill 提炼输入
- `architecture/` 与 `adr/`：架构和决策文档

## 当前执行约定

从当前迭代开始：

- 继续开发新功能时，同步维护文档
- 不把“补文档”当成独立的遥远阶段
- 在每一轮交付说明里，默认检查文档是否需要同步更新
- 每次开始新会话，先读 `AI-Hub-开发路线图.md`
- 每次结束新会话，确认是否需要更新 `AI-Hub-工作流沉淀与Skill固化指南.md`
